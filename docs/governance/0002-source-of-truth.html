Source of Truth

Date: 2026-05-27

This page is the standing contract for the project. If future work conflicts with this page, this page wins until William explicitly changes it.

North Star

Build a system that turns one uploaded book into a clean, organized, evidence-backed learning engine with a knowledge graph. The first version must prove the extraction foundation before trying to automate many books.

Priority Order

  1. Phase 1: complete, clean, organized PDF extraction.
  2. Phase 1 auditability: every cleanup and omission is inspectable.
  3. Canonical document structure: pages, sections, objects, reading order, and provenance.
  4. Only then: retrieval, reasoning, claims, entities, relations, and graph construction.
  5. Only after quality exists: automation and broader multi-book support.

Hard Rules

  1. Do not optimize for a chatbot before the extracted book is clean.
  2. Do not let retrieval hide extraction failures.
  3. Do not silently drop content from a PDF.
  4. Do not treat LLM output as ground truth without provenance and review.
  5. Do not widen to many books before one book passes Phase 1 gates.
  6. Do not spend expensive model calls without recording why the model was chosen.
  7. Do not use Markdown as the durable project documentation format.
  8. Do not adopt a paragraph merge policy merely because heuristic warning counts improve.
  9. Stage, commit, and push meaningful checkpoints when the repo state should be preserved.

Candidate-First Rule

No Phase 1 output should be called canonical until it has passed an audit step. The first extraction and classification pass produces candidates. Canonical book content may only be produced after audit, comparison, and review.

Acceptable audit evidence can include deterministic validation, page-image inspection, OCR comparison, model-assisted review, or human approval. Models may classify, link, order, and suggest repairs from preserved evidence. They may not invent, rewrite, or fill missing book content.

Canonical Promotion Rule

Canonical does not mean perfect. It means evidence-bound, audited enough under current rules, and safe to use downstream. Promotion must produce a log of what was promoted, what was blocked, and why.

The first promotion gate is paragraph-only. Structure, page artifacts, unknown objects, OCR, model review, retrieval, and knowledge graph work remain outside this gate until explicitly designed and audited.

Gold Review Rule

Paragraph merge policy changes need a verified answer key. A merge policy cannot become active merely because heuristic warnings improve; it must improve against the authoritative gold review set and must not worsen audit safety metrics such as over-splitting, missing lines, structure leakage, artifact leakage, or downstream safety.

Gold review files live under reviews/<book_id>/gold/. Seed or example rows are useful for shaping the workflow, but they are not scoring truth until explicitly marked authoritative after visual review.

When a cross-page continuation experiment improves gold scores, it still cannot become active until proposed joins are reviewed. The join review must separate gold-covered joins from unscored joins and surface boundary, structure, and long-paragraph risks. Durable decisions for those joins belong in reviews/<book_id>/cross_page_join_decisions.jsonl, not in generated run output.

For Douglass phase1_v3, v2_cross_page_continuation is now the active paragraph merge policy because it passed the gold, side-effect review, blocker investigation, and validation gates. This adoption does not unlock downstream intelligence; the canonical paragraph review gate still controls whether retrieval, embeddings, reasoning, or graph work may depend on the output.

The post-adoption safety gate currently identifies bbox_span_risk as the next dominant blocker. That blocker must be diagnosed from evidence before changing extraction rules again. The current diagnosis classifies 23 cases as likely true paragraph grouping defects and 8 as needing visual review after the gold-set gap queue was expanded to 0. The front-matter/metadata review found 7 valid Preface or Letter paragraphs and 2 valid Chapter I body paragraphs caught by an overbroad early-page warning. Do not change promotion or grouping rules until the visual-review queue is complete.

Phase 1 Evidence Stack

Phase 1 is a truth filter before it is a cleaning tool. Every extracted object should remain traceable through raw PDF evidence, deterministic layout objects, candidate bucket assignment, visual page evidence, bounding-box evidence when available, review overrides when applied, and audit validation.

The first reusable extraction method must expose this stack before it asks retrieval, reasoning, OCR, or models to trust the output. The current candidate streams are main_paragraph_candidates.jsonl, structure_candidates.jsonl, page_artifacts_candidates.jsonl, and unknown_objects.jsonl.

Review Override Rule

Reviewer corrections must be stored as tracked explicit artifacts, not hidden inside detector changes. A review override can correct a candidate bucket while preserving the original assignment, corrected assignment, evidence reference, reviewer, reason, and date. Overrides remain candidate-level until canonical promotion. Generated runs may record applied overrides, but durable review truth lives under reviews/.

Documentation Truth

Durable project memory lives in docs/ as HTML. Machine-readable guidance lives in docs/governance/codex_guidance.json. Generated book runs live under data/runs/ and must not be confused with source code.

Exception: README.md may exist as a GitHub-facing orientation adapter because GitHub repository landing pages render Markdown better than local HTML. It is not the durable source of truth; it should point readers back to the HTML documentation.

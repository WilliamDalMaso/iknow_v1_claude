Formatting Rules

Date: 2026-05-27

These rules capture the formatting decisions made during the first project setup. They are meant to keep the repo readable and prevent visual drift.

Local HTML Documentation

Local durable documentation in docs/ uses the standard light document style. Do not use dark mode for ordinary project docs.
  1. Use #fbfbf8 for the page background.
  2. Use #17202a for primary body text.
  3. Use #135f7f for links.
  4. Use #d8d6cc for section dividers.
  5. Use #ece9df for inline code backgrounds.
  6. Use a constrained main column around 920px to 980px.
  7. Use generous page padding, currently 48px 24px.
  8. Use simple headings, paragraphs, ordered lists, and tables. Avoid decorative UI.

Callouts

  1. Use green-left-border callouts for principles or source-of-truth rules.
  2. Use warning callouts sparingly for risks, not decoration.
  3. Do not turn every section into a card.

Observability Dashboard

Dark mode is reserved for runtime observability surfaces such as the localhost dashboard. It is not the default style for project documentation.
  1. Keep the observability dashboard black and white.
  2. Use it for live process state, build events, and runtime events.
  3. Keep runtime logs ignored by git unless a curated finding becomes documentation.

GitHub Reader Experience

  1. README.md exists because GitHub renders Markdown best on the repo landing page.
  2. The README is an adapter, not the local source of truth.
  3. When README content changes project meaning, update the HTML source of truth first.

Future Formatting Rule

If a future page needs a different visual treatment, document why before making the exception. The default assumption remains: local docs are light, readable, stable, and restrained.

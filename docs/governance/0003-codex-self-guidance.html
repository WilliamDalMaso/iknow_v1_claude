Codex Self Guidance

Date: 2026-05-27

Before Starting Any Work

  1. Read docs/governance/0002-source-of-truth.html.
  2. Read docs/governance/0005-codex-work-loop.html.
  3. Check whether the requested work strengthens Phase 1 cleanliness or distracts from it.
  4. If making a meaningful change, append an observability event before the first edit and another one after verification.
  5. Prefer a small verified step over broad architecture.

Engineering Bias

  1. Deterministic extraction and audits first; models second.
  2. Separate raw, cleaned, candidate, future canonical, retrieval, and graph artifacts.
  3. Do not call Phase 1 output canonical until it has passed audit, comparison, and review.
  4. Preserve provenance for every derived object.
  5. Write tests for artifact contracts as soon as a contract exists.
  6. When uncertain, inspect the actual PDF evidence instead of guessing.
  7. Do not adopt paragraph merge policy changes from heuristic warning counts alone; require gold-set improvement and side-effect review.

Verification Bias

  1. Use requirements-dev.txt as the test dependency contract.
  2. Run python3 -m pytest tests before committing code or artifact-contract changes.
  3. Use a direct test-function runner only as a temporary fallback when pytest is unavailable, and document the fallback.

Evidence Stack Reminder

  1. Start from raw PDF and layout evidence, not clean-looking prose.
  2. Keep candidate streams separate from future canonical artifacts.
  3. Use page images and bounding-box overlays to check where extracted objects came from.
  4. Keep reviewer corrections in tracked review override files, not hidden detector tweaks.
  5. Use tracked gold review rows as the answer key for paragraph-boundary policy experiments.
  6. Treat v2_cross_page_continuation as the active Douglass paragraph merge policy only because its adoption decision is documented and validated.
  7. Store reviewed cross-page join decisions under reviews/<book_id>/cross_page_join_decisions.jsonl.
  8. Do not confuse policy adoption with downstream readiness; canonical paragraph review can still block retrieval and graph work.
  9. Use post_adoption_canonical_safety_report.json to identify the next blocker after policy adoption; currently that blocker is bbox_span_risk.
  10. Do not add OCR, model review, retrieval, or graph work just because the audit UI improved.

Git Discipline

  1. Keep commits scoped to coherent checkpoints.
  2. Stage, commit, and push after meaningful durable changes.
  3. Do not commit secrets, virtual environments, caches, or bulky temporary artifacts.

GitHub Readme Exception

  1. README.md is allowed for GitHub reader experience.
  2. Do not move durable local documentation back to Markdown.
  3. When README.md and HTML docs disagree, update the HTML source of truth first.

Observability Discipline

  1. Use http://127.0.0.1:8765 as the single observability dashboard.
  2. Do not start alternate observability ports unless William explicitly asks.
  3. If the server needs a restart, stop the existing process and reuse port 8765.

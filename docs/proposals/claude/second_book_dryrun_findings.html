Second-Book Dry-Run Findings — The Hobbit

Author: Claude · Date: 2026-05-28 · Phase: C reconnaissance

Read-only reconnaissance. Phase 1 was run once on data/books/the_hobbit.pdf into a disposable, git-ignored candidate run. Nothing was promoted, no policy was adopted, no gold set was created, and Douglass was untouched. The run directory was deleted after capturing these findings. This is diagnosis, not widening to a second book. To reproduce: python3 src/phase1_extract.py data/books/the_hobbit.pdf --book-id the_hobbit --run-id dryrun.

Headline

The pipeline ran end-to-end on a brand-new 270-page book and produced clean text — the core extraction generalizes. But the review / validation / policy layer is hardcoded to Douglass, so the run cannot be judged or trusted for a second book yet. The validation report came back fail — for the right reason (Douglass-specific checks), which is exactly the signal we wanted.

Where the single-book assumptions live (the real findings)

  1. Validation conflates generic structure checks with Douglass acceptance criteria. The failing checks are all Douglass-specific assertions run against every book: guarded_chained_experiment_keeps_cp_000103_fixed (a specific Douglass paragraph id), guarded_chained_experiment_blocks_rejected_join (the Douglass pages 59–61 false join), and the guarded_policy_adoption_* gates. The Hobbit has no cp_000103, so it can never pass. Fix: split generic structural validation from per-book acceptance checks.
  2. Page-furniture detection is literally Douglass text. looks_like_running_page_furniture matches only narrative | life of frederick douglass. The Hobbit's running headers / chapter titles will not be recognized or stripped. Fix: per-book furniture patterns (config-driven).
  3. A Douglass-gold-validated paragraph policy is applied globally. v3_chained_cross_page_continuation_guarded was tuned and gold-validated for Douglass only; it is the active policy for The Hobbit by default. Fix: policy selection per book.
  4. The Hobbit contains layouts Douglass never had — notably verse/songs (e.g. “Far over the misty mountains grim / To dungeons deep and caverns dim”), plus heavy dialogue. The paragraph-continuation heuristics were never designed for poetry and will mis-merge it. Fix: needs new heuristics and, first, a gold set to measure against.
  5. No gold set exists for The Hobbit, so quality is currently unmeasurable. Per the gold-gated method, warning counts alone are not truth. Fix: authoring a gold set is a prerequisite before any second-book quality claim.
  6. Minor: Douglass-specific ids/strings are baked into report builders (e.g. cp_000103, douglass_narrative:p0107…). These belong in per-book data, not in shared code.

Implications for the roadmap

  • Phase C is now concretely scoped. The book-specific surface area is: (a) the active paragraph policy, (b) furniture patterns, (c) the acceptance/validation checks, (d) curated decisions + gold. Everything else generalized cleanly.
  • This also guides the remaining Phase B decomposition. Book-specificity concentrates in the review / validation / gold / policy-adoption code, so those modules should be designed to take a per-book config object rather than hardcode Douglass.
  • Gates still hold. The hard rule stands: no second-book adoption until Douglass passes its gates and The Hobbit has its own gold set. This dry-run changes none of that.

Recommended next step

Introduce a small per-book config (policy + furniture patterns + paths) and split generic validation from Douglass acceptance checks — the first concrete Phase C increment. This is additive and does not change Douglass behavior (the regression test will confirm).

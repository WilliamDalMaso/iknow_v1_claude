Douglass Gate-Closing Plan

Author: Claude · Date: 2026-05-28 · Step: 1 (diagnosis & plan only)

Planning only — no behavior change. This reads the existing diagnostic artifacts and turns them into a concrete plan to flip safe_for_downstream from false to true, which is what unblocks mapping, embedding, and graph/intelligence. Implementing the fixes (Steps 2–4) changes extraction behavior and needs explicit authorization plus the gold/review gates.

The goal

Today Douglass phase1_v3 is safe_for_downstream: false with 115 promoted canonical paragraphs, 72 risky, and 129 warnings. The single decision that opens the downstream phase is: resolve the blocking defects, confirm no audit-safety regression, and flip the flag.

What is already closed vs. what blocks

The post-adoption remediation plan sorts the 39 bbox/span cases into four queues. Three are done; one blocks.

So the gate hinges on one group of 23 defects. The remediation plan's recommended_order is exactly: “Review and design a narrow correction path for the 23 likely true paragraph grouping defects under the active policy.” The design scaffold (narrow_grouping_correction_design.json) exists but is empty (confirmed_defects: 0, designs: []) — the visual confirmation and correction design were never completed. That is the gap.

What the 23 defects actually are

They are accidental paragraph merges: two (or more) real paragraphs got combined into one canonical paragraph, so its source lines span a suspicious vertical distance on the page (source_lines_span_suspicious_distance, existing_bbox_likely_cause: true_accidental_merge). Two confirmed-looking examples:

cp_000025 (p21): 12 source lines, spanning ~27% of page height. Starts “Called thus suddenly away…” and ends “master and father.” — two distinct paragraphs joined.
cp_000032 (p29): 15 source lines, ~34% of page height. Ends with “words:—”, which is the lead-in to a quoted song — a textbook paragraph-break that was merged through.

Affected pages: 21, 29, 30, 33, 37, 41, 42, 45, 46, 50, 73, 74, 84, 88, 89, 92, 93, 95, 97, 98, 112, 118.

Why this is the gate. A merged paragraph becomes one oversized chunk. If we embedded or graphed it as-is, two unrelated ideas would share one vector / node — quietly corrupting retrieval and reasoning. This is precisely the failure the “clean first” rule exists to prevent, so the project correctly refuses to proceed until it is fixed.

The fix class (from the remediation plan)

A narrow, guarded paragraph-split correction — split these specific over-merged candidates at their true internal paragraph boundary. Explicitly not a broad merge-policy rewrite (the plan warns: “do not apply one broad merge fix to all remaining cases”). The split signal is likely already computed: large intra-candidate vertical gap (line_gap), indentation of the next line (starts_new_indented_paragraph), and a terminal boundary before it (terminal_line_boundary). The work is to define a conservative rule from those signals that splits the true defects without over-splitting clean paragraphs.

The plan to close the gate

  1. Confirm (visual, review-only). Inspect the page images for the 23 candidates and label each a true defect or a false alarm. Populate narrow_grouping_correction_design.json with the confirmed pattern(s). No behavior change.
  2. Author gold rows. Add authoritative gold paragraph-boundary rows for the confirmed pages, so a split correction can be scored rather than guessed. (Gold lives under reviews/douglass_narrative/gold/.) No behavior change.
  3. Design + implement the guarded split. Encode the conservative split rule. This changes extraction behavior → needs explicit authorization.
  4. Prove it at the gates. Show it (a) improves against the gold rows, (b) does not regress audit-safety metrics (warnings, bbox-span counts must not worsen), (c) passes side-effect review (every newly created split inspected). Record an adoption decision.
  5. Re-run, re-verify, flip. Re-run Douglass, confirm the regression fingerprint still matches except the intended splits, and update safe_for_downstream to true with the evidence.
  6. Then the downstream phase legitimately opens: mapping → embedding → graph/intelligence.

Honest effort estimate

  • Steps 1–2 (confirm + gold): the bulk of the human/judgment effort — 23 cases to eyeball, plus authoring gold boundaries for ~22 pages. Methodical, not hard.
  • Step 3 (the rule): likely a small, conservative addition to the existing paragraph-segmentation logic, since the needed signals (gap, indent, terminal punctuation) are already present.
  • Steps 4–5 (gates + flip): mechanical once the rule holds, and fully guarded by the regression test and gold evaluation.
  • Risk: the main risk is over-splitting clean paragraphs. The gold set + side-effect review are exactly what contain that risk.

Authorization boundary

Steps 1–2 are review/proposal work I can do in-lane without changing behavior. Step 3 onward edits extraction behavior and must clear the gold/side-effect gates — so it needs an explicit go-ahead. Nothing here relaxes any gate or widens to a second book.

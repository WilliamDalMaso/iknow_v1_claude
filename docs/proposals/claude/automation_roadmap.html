Claude Lane — Automation Roadmap

Author: Claude · Date: 2026-05-28

Goal: a trustworthy, book-agnostic, repeatable automated extraction and cleaned-text pipeline.

Key fact: the pipeline already exists end-to-end. run_phase1(pdf_path, book_id, run_id) in src/phase1_extract.py runs PDF → raw → cleaned text → review/audit in one shot, with a CLI main() and validate_phase1_run(). The work is not "build a pipeline"; it is to make that pipeline verifiable, book-agnostic, and repeatable without bypassing the gold/review gates.

Through-line

The automated deliverable is candidate cleaned text (clean_objects.jsonl + cleanup_log.jsonl), produced for any book by one command, with a machine-readable report stating what is blocked and whether it is promotable. Automation means orchestration plus honest gate reporting — never auto-promotion. All existing blocks stay.

Phases

EMake it repeatable Single entry command + CI-style fixture test on a tiny PDF; then the Hobbit diagnostic dry-run (extraction + cleaned text + audit, no promotion). Planned

Critical path

A → B → C. Trustworthy baseline, safe-to-change code, then book-agnostic orchestration. C is where "automated extraction + cleaned text for any book" actually lands.

Phase A delivered in this pass: src/baseline_manifest.py (compute/verify tool + CLI), the committed baseline under docs/baselines/douglass_narrative/phase1_v3/ (164-file manifest + snapshot of the decision-bearing reports), and tests/test_baseline_manifest.py. Verified MATCH against the local run.

What stays fixed throughout

  • Automation never unblocks downstream work; promotion stays gold-gated, side-effect-reviewed, and human-adopted.
  • Douglass stays safe_for_downstream: false until its 23 grouping defects and 39 bbox/span cases are resolved.
  • A second book stays a candidate, not canonical.

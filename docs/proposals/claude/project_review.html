Claude Lane — Project Review

Author: Claude · Date: 2026-05-28 · Lane: iknow_v1_claude (review/proposal fork of iknow_v1)

This is a review-only document. It proposes; it does not change extraction behavior, promote structure, or unblock downstream work. All recommendations respect the project's evidence-first, candidate-first, gold-gated method.

1. What This Project Is

A one-book-first Phase 1 extraction engine that turns a PDF into clean, audited, evidence-bound document objects. The active book is douglass_narrative. Downstream retrieval, embeddings, reasoning, and graph work are intentionally blocked until Phase 1 canonical paragraph review is safe. The method is: candidate-first → gold-gated → side-effect-reviewed → formally adopted.

3. Architecture

  • src/phase1_extract.py — 7,524 lines, 106 top-level functions in a single module. It owns IO helpers, layout segmentation, furniture detection, cross-page join logic, promotion, review reporting, gold evaluation, bbox/span diagnosis, and HTML audit rendering.
  • src/observability_server.py (390 lines) — dependency-free localhost dashboard on port 8765.
  • src/observe.py (48 lines) — CLI event appender writing to data/runs/observability/events.jsonl.
  • src/config.py (39 lines) — env/model config; model names intentionally not defaulted.
  • tests/test_phase1_extract.py (48KB) — the bulk of the 27 passing tests.
  • reviews/<book_id>/ — tracked curated decisions (gold sets, overrides, join decisions).
  • data/runs/<book_id>/phase1_vN/ — generated artifacts; git-ignored.

4. Strengths

  • Genuinely evidence-first: every canonical object is gated behind audit, gold, and side-effect review. The discipline is real, not decorative.
  • Durable curated decisions (overrides, join decisions, gold) live under tracked reviews/, separate from detector logic — corrections are not hidden in code.
  • Strong governance: hard rules, a machine-readable guidance file, and a documented work loop.
  • Tests pass fast (27 in ~0.2s) and cover artifact contracts.
  • Honest status reporting: the project refuses to call itself safe_for_downstream while real grouping defects remain.

5. Top Risks

R1 — The canonical state is untracked. .gitignore excludes data/runs/* and data/books/*.pdf. The entire Douglass v3 run that governance describes (115 paragraphs, all reports) exists only on this machine. A fresh clone cannot reproduce or verify the documented state. See the consistency risk report.
  • R2 — Monolith. 7.5k lines / 106 functions in one file makes review, reuse, and second-book generalization hard and raises regression risk on any edit.
  • R3 — Book-coupling. Gold sets, join decisions, and several documented numbers are specific to douglass_narrative. Second-book readiness is unproven.
  • R4 — Doc/state drift. Hard-coded statistics in governance prose (counts, page numbers) can silently diverge from regenerated runs.
  • R5 — Observability is local-only and ephemeral (single port, jsonl on disk, git-ignored), so the audit trail of how decisions were made is not durable.

6. What Should Stay Blocked

No change here — I endorse the existing blocks. While safe_for_downstream is false, keep blocked: structure promotion, page-artifact promotion, unknown-object promotion, OCR, AI/model review, embeddings, retrieval, and graph work. Also keep the second book blocked from Phase 1 adoption until its own gold set exists (a dry-run extraction for diagnosis is a separate, lower-risk question — see the second-book readiness plan).

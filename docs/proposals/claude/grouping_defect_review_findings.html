Grouping-Defect Review Findings (Gate-Closing Step 2)

Author: Claude · Date: 2026-05-29 · Mode: review-only (no extraction behavior changed)

Headline reframing. The 23 cases the diagnostics labeled "likely-true paragraph grouping defects" are, on inspection, not defects — they are valid, correctly bounded long paragraphs that the bbox/vertical-span heuristic over-flags. Douglass writes long paragraphs; the heuristic treats large vertical span as a probable accidental merge, which is a false signal for this text. Splitting them would damage correct output.

What I did

  • Visually inspected 6 of the 23 against the rendered page images, chosen to cover the riskiest shapes: the largest spans (cp_000056 p46, 19 lines/43%), a two-flagged-paragraph page (cp_000045/cp_000046 p41), and the quote/document lead-in endings (cp_000032 p29 song; cp_000099 p112 certificate). All six are single, indented, correctly terminated paragraphs.
  • 1 more (cp_000034 p30) was already covered by an existing authoritative gold row as a single paragraph.
  • Text-triage of all 23: 20 end in terminal punctuation (a complete sentence); the other 3 end in ":—" (lead-ins to a song/document that are correctly excluded from the paragraph). Consistent with "long but valid".
  • Authored 5 new authoritative gold rows (reviewer claude_visual_audit) certifying the verified paragraphs as single. The gold set grew 17 → 22 authoritative rows.
  • Recorded all 23 adjudications in reviews/douglass_narrative/grouping_defect_review.jsonl.

Evidence it is correct (not a guess)

After adding the gold rows, the gold evaluation reports 22/22 paragraphs matched, precision = recall = 1.000, zero over-merges, zero over-splits. The extraction policy already groups these exactly as the gold says they should be — formal confirmation that the flagged paragraphs are correct and the warnings are false alarms.

What this means for the gate

The original remediation plan assumed a "narrow grouping correction" (a paragraph split). The visual evidence says that would be wrong. The corrected path to safe_for_downstream:

  1. Finish visually confirming the remaining 16 and add gold rows (review-only).
  2. Refine the audit/risk heuristic so a long-but-gold-confirmed paragraph is not counted as "risky" — this is a review-side change, not an extraction change, and must itself be gold-gated and side-effect reviewed (needs explicit authorization).
  3. Re-evaluate safe_for_downstream once the risk count reflects real defects (which so far appear to be none in this cluster).

Net: the gate looks much closer than "23 defects" implied — the canonical paragraphs in this cluster appear correct; the blocker is heuristic over-sensitivity, not bad extraction. This must be confirmed across the remaining 16 before declaring the cluster clean.

Provenance & safety

  • No extraction behavior changed. Re-running Douglass after the gold edit changed only review/scoring artifacts (gold evaluation, policy-decision reports, audit HTML); every extraction artifact was byte-identical (verified via the regression fingerprint).
  • The regression baseline was deliberately regenerated to absorb the intentional gold-input change, and the delta was confirmed to be review-only.

Grouping-Defect Review Findings (Gate-Closing Step 2)

Author: Claude · Date: 2026-05-29 · Mode: review-only (no extraction behavior changed)

Headline reframing. The 23 cases the diagnostics labeled "likely-true paragraph grouping defects" are, on inspection, not defects — they are valid, correctly bounded long paragraphs that the bbox/vertical-span heuristic over-flags. Douglass writes long paragraphs; the heuristic treats large vertical span as a probable accidental merge, which is a false signal for this text. Splitting them would damage correct output.

What I did

  • Visually inspected all 22 single-page cases against the rendered page images (pages 21, 29, 30*, 33, 37, 41×2, 42, 45, 46, 50, 73, 74, 84, 88, 89, 92, 93, 95, 97, 98, 112, 118), including the largest spans (cp_000056 p46 and cp_000090 p93, ~19–20 lines) and every quote/document lead-in (song p29, apostrophe p74, certificate p112). Every one is a single, correctly bounded paragraph. (*p30 / cp_000034 was already covered by existing authoritative gold.)
  • 1 case is a cross-page tail (cp_000082 p88): the page-88 continuation of a paragraph beginning on p87 — valid, no internal over-merge. Left out of gold pending the separate cross-page boundary question.
  • Authored 20 authoritative gold rows total (reviewer claude_visual_audit) across the two passes, certifying the verified paragraphs as single. The gold set grew 17 → 37 authoritative rows.
  • Recorded all 23 adjudications in reviews/douglass_narrative/grouping_defect_review.jsonl.

Evidence it is correct (not a guess)

After adding the gold rows, the gold evaluation reports 37/37 paragraphs matched, precision = recall = 1.000, zero over-merges, zero over-splits. The extraction policy already groups these exactly as the gold says they should be — formal confirmation that the flagged paragraphs are correct and the warnings are false alarms.

What this means for the gate

The original remediation plan assumed a "narrow grouping correction" (a paragraph split). The visual evidence says that would be wrong. The corrected path to safe_for_downstream:

  1. [DONE] Visually confirm the cluster and add gold rows (review-only) — all 23 reviewed; gold set 17 → 37; evaluation 37/37 matched at precision = recall = 1.000.
  2. Refine the audit/risk heuristic so a long, gold-confirmed paragraph is not counted as "risky" — this is a review-side change, not an extraction change, and must itself be gold-gated and side-effect reviewed (needs explicit authorization).
  3. Re-evaluate safe_for_downstream once the risk count reflects real defects (which, in this cluster, are none).

Net: the gate is much closer than "23 defects" implied — the canonical paragraphs in this cluster are confirmed correct; the blocker is heuristic over-sensitivity, not bad extraction. The next, separately-authorized step is the review-side heuristic refinement, after which safe_for_downstream can be reconsidered. (Note: other warning categories outside this 39-case bbox/span cluster — e.g. front-matter/chapter-boundary risks — would also need a pass before a full downstream-safe declaration.)

Self Review: Cracks and Next Risks

Date: 2026-05-27

This review checks whether the current project setup is disciplined enough to build the engine we actually want. The answer is: directionally yes, but only if Phase 1 remains gated and we keep documenting failures before adding downstream intelligence.

What Is Solid So Far

  1. The source of truth puts clean extraction before retrieval, reasoning, and graph construction.
  2. The old-project mistakes were reviewed and converted into rules instead of copied blindly.
  3. The repo has a durable HTML documentation spine, a GitHub README adapter, and machine-readable guidance.
  4. The first Phase 1 run produced raw, clean, page inventory, object, canonical, cleanup, and audit artifacts.
  5. Observability now uses one fixed endpoint: http://127.0.0.1:8765.
  6. The first book choice is conservative: douglass_narrative is small enough to audit but not trivial.

Cracks

CrackRiskCorrection
Observability is not yet supervised The dashboard can stop between sessions and must be restarted manually. Keep port 8765. Later add a local launcher or health check, but do not create new ports.
Phase 1 v0 is line-based Line objects are audit-friendly but not clean document objects. Downstream use would be premature. Next slice must build paragraph/object segmentation before any retrieval work.
No visual/OCR inspection yet Image-only page 2 is flagged but not understood. Content could be missed. Add page-image rendering or OCR review for flagged image/no-text pages.
Heading detection is provisional False headings can contaminate canonical structure. Separate heading candidates from accepted structure until audited.
No schema-level artifact validator yet Future extractor changes could silently break artifact contracts. Add a contract validator that checks required files, row counts, fields, and page coverage.
No dependency/project environment file yet Reproducibility depends on ambient Python packages such as pdfplumber. Add dependency management before the extractor grows much further.
Runtime artifacts are ignored by git Good for repo hygiene, but important run findings can disappear if not curated. Promote meaningful findings into docs/history or docs/lessons immediately.
Books are local-only Remote clone cannot reproduce the exact run without local source PDFs. Keep PDFs local for now, but record source manifest hashes and document local setup expectations.

Model Judgment

No model should be used as the extractor of record at this point. The current best use of models would be bounded review after deterministic artifacts exist: classify flagged failures, compare cleaned objects against raw text, or suggest repair options for specific noisy spans. Using a model earlier would make the system look smarter while weakening auditability.

Competence Check

Codex is useful enough to build this engine if it remains constrained by contracts, audits, and human-readable evidence. Codex is not reliable enough to simply "understand the PDF" and produce trusted clean content without deterministic artifacts and review gates. The build strategy should therefore keep Codex as an engineering agent, not as an unquestioned document authority.

Next Correct Move

Build Phase 1 v1 paragraph/object segmentation and artifact validation. Do not start retrieval, embeddings, claims, entities, or graph work yet.

0002 — Phase A: Reproducibility Baseline + Manifest Tool

When: 2026-05-28 (commit ac1d9f9) · Tests after: 31 passed

Why

The biggest risk found in the project review (R1): the "finished" Douglass phase1_v3 run — the 115 canonical paragraphs and all reports the governance numbers cite — lived only under git-ignored data/runs/. A fresh clone could neither reproduce nor verify it. Without a trustworthy, verifiable baseline, neither automation nor refactoring rests on solid ground.

What I did

  • Wrote src/baseline_manifest.py — a tool that records sha256 + size for every file in a run directory, and verifies a later run against it (CLI: compute / verify). It only hashes bytes; it changes no extraction behavior.
  • Committed a baseline under docs/baselines/douglass_narrative/phase1_v3/: a 164-file manifest, plus a snapshot of the small decision-bearing report JSONs (carrying warning_count: 129, safe_for_downstream: false), plus an index.html explaining how to verify.
  • Added tests/test_baseline_manifest.py — self-contained contract tests (no dependency on local-only run data) covering match / changed / missing / unexpected.
  • Wrote docs/proposals/claude/automation_roadmap.html framing the A→E plan.

How — key insight

The pipeline already had a one-shot entry point: run_phase1(pdf, book_id, run_id) with a CLI. So the goal was never "build a pipeline" — it was to make the existing one verifiable, book-agnostic, repeatable. Phase A delivered "verifiable." 

python3 src/baseline_manifest.py compute \
  data/runs/douglass_narrative/phase1_v3 \
  docs/baselines/douglass_narrative/phase1_v3/artifact_manifest.json
# -> wrote 164 entries; verify -> MATCH

Result

R1 closed: the documented run is now verifiable from version control (without committing raw PDFs or bulky output). pytest 31 passed; manifest + guidance JSON valid; diff clean.

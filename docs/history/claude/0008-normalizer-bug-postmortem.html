0008 — Failure Post-Mortem: Regression-Normalizer Collision Bug

When: 2026-05-29 (fixed in commit 3724a13) · Tests after fix: 44 passed

Symptom

After the docs-only gate-closing-plan commit (c7a2e54), the suite showed 1 failed, 41 passed — test_phase1_regression.py reported changed=['phase1_audit.html']. No production code had been touched. I also realized I had committed c7a2e54 without re-running the suite first, so the red state was pushed.

Investigation (what I ruled out, in order)

  1. Intermittent nondeterminism? Ran Douglass 6× and hashed the normalized audit HTML → one identical hash. Not flaky.
  2. Did the decomposition change output? Ran the pre-decomposition code (git show 5e7b775:…) vs current → byte-identical audit HTML. Code was not the cause.
  3. Did the normalizer change? Diffed baseline_manifest.py at the baseline commit vs now → identical.
  4. Was it run-id dependent? Re-ran into the exact baseline run-id (phase1_v3_repro) → still mismatched the committed hash.
  5. Two fresh runs under the real normalizer (canonical_content) → they differed, and the diff showed the smoking gun: heading_candidate had become heading<run_id>ndidate for a run named _ca.

Root cause

The regression normalizer — not the pipeline — was buggy. _neutralize_text did a bare text.replace(run_dir.name, "<run_id>"). When the run-id is a substring of real content (a run named ca inside the word "candidate"), it corrupted the text. The committed golden baseline had been captured with such a corrupted phase1_audit.html hash, so a clean re-run no longer matched. The extraction pipeline itself was never wrong.

Why it passed before but failed now

The data artifacts (JSON/JSONL) were always correct, so only the audit-HTML hash was ever in question. The mismatch surfaced when a run-id happened to collide with content; the earlier-passing runs used non-colliding ids. Regenerating the baseline from a clean run made the one stale hash explicit and fixable.

The fix

  • Changed _neutralize_text to neutralize the run-id only as a path segment ("/" + run_id), never as a bare substring.
  • Proved run-id independence: runs named _ca, _cb, _regression_check, phase1_v3_repro, candidate_run now all produce one identical audit-HTML hash.
  • Regenerated normalized_manifest.json from a clean run — only the phase1_audit.html hash moved; the other 162 fingerprints were already correct, confirming data determinism was never in question.
  • Added two guard tests: a run-id colliding with content must not corrupt it; a run-id appearing as a path segment must be neutralized.

Result

Repo green again: pytest 44 passed, stable across repeated runs; regression passing.

Lessons recorded

  • Run the full suite before committing — even docs-only commits — so a red state is never pushed.
  • String normalization must use boundary-aware replacement, never bare substring replace.
  • A golden baseline is only as trustworthy as the normalizer that built it; test the normalizer's edge cases (collisions) explicitly.
  • Reassurance: the pipeline's determinism held throughout; this was a tooling bug, contained and fixed.

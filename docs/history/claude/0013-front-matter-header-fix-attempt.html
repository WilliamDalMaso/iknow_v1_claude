0013 — Front-Matter Header-Leak Fix: Attempted, Gated, Reverted

When: 2026-05-30 · Outcome: not adopted (gates caught a large side-effect); working tree restored, suite 45, regression MATCH.

Why

William authorized fixing the 3 front-matter content defects (entry 0010). This entry covers the first attempt: the two running-header leaks (cp_000009 "PREFACE xiii", cp_000010 "xiv PREFACE").

The fix (correct in isolation)

Root cause: classify_line only treats a header as furniture when its uppercase ratio ≥ 0.75; lowercase roman page numbers ("xiii"/"xiv") on pages past the page≤12 front-matter window drop it under threshold, so the header is classed paragraph_line and merged into the body. I added a minimal running-header rule (short line + page-number token at an edge + all-uppercase title) placed after the existing heading checks, so it only affects lines that would otherwise be body.

It worked surgically: "PREFACE xiii"/"xiv PREFACE" became page_artifacts, the leaked text was removed, and all 229 body-narrative paragraphs (pages ≥19) were byte-identical. Gold evaluation stayed 37/37, precision = recall = 1.000.

Why it was NOT adopted

Removing the page-13 header let "also of those who committed…" rejoin its page-12 paragraph — a new, correct cross-page join. But the curated cross-page join decisions (reviews/douglass_narrative/cross_page_join_decisions.jsonl) are keyed by positional join_id (xpage_join_0001…). Inserting a join early renumbers every later join, so 28 of 31 curated decisions no longer matched their join and validation went pass → fail (cross_page_join_decision_candidate_ids_match). The body text was fine; the review-decision keying was destabilized.

Per the evidence-first method, a change that regresses validation is not force-landed. I reverted the code, removed the trial run, and confirmed the lane is clean (suite 45, regression MATCH).

The real finding (architectural)

The curated cross-page join decisions are keyed by a positional id that is not stable under any change to the join set. This is a latent fragility: any future extraction change that adds/removes a join renumbers the decisions and breaks validation. It is worth making join_id content-stable (derive it from the joined candidates' page + text signature) independently of this header fix.

Options to actually land the header fix

  1. Stabilize join_id first (content-based), re-key the 31 curated decisions once, then the header fix simply adds one new front-matter join to review. Foundational; removes the fragility for all future changes. (Recommended, but a separate authorized change.)
  2. Scope front matter out of the downstream substrate — the leaked headers live in front matter (prefaces by others) that may not belong in book intelligence anyway; this sidesteps the fix entirely.
  3. Re-curate the 31 positional join decisions to the post-fix set — works, but brittle (breaks again on the next join-set change). Not recommended.
  4. Defer — the leak is confined to 2 front-matter paragraphs; leave it documented.

Note: the digitization-metadata defect (cp_000003, page 4) was not attempted here; it does not touch the join set and can be handled independently.

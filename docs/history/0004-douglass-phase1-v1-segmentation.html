Douglass Phase 1 v1 Segmentation

Date: 2026-05-27

What Changed

  1. src/phase1_extract.py now defaults to phase1_v1.
  2. Line-level prose is merged into paragraph objects while preserving source line ids.
  3. Detected headings are written as heading_candidate, not trusted canonical headings.
  4. Paragraph splitting uses deterministic line geometry when available, especially first-line indentation.
  5. Each run emits validation_report.json and the audit page shows validation status.

Method Detail

The v1 segmenter uses pdfplumber.extract_text_lines() when available. In the Douglass PDF, continuation lines often begin around x0=43.3, while new paragraphs often begin around x0=64.9. V1 uses that indentation gap to split paragraph candidates.

What Went Right

  1. V1 preserves raw page text and source line ids while creating higher-level paragraph objects.
  2. The validator caught a real ordering bug: it originally checked for the audit page before the audit page was written.
  3. The validator now checks that validation_report.json itself exists.
  4. The paragraph count and large-block count both moved in the right direction.

What Is Still Bad Or Risky

V1 is better than v0, but it is not final clean extraction.
  1. 35 paragraph candidates are still over 300 words.
  2. Some long objects may be true long paragraphs, but some may still be page-spanning over-merges.
  3. Running headers such as roman numeral plus section title are not yet cleanly separated from structure.
  4. Image-only page 2 is still only flagged, not visually/OCR inspected.
  5. No model has reviewed flagged cases yet, which is correct for now but will become useful later.

Next Correct Move

Add Phase 1 quality diagnostics for suspicious paragraph objects, especially overlong paragraph candidates, running headers, and image-only pages. Do not move to retrieval yet.

Executive Judgment

The previous project had a strong habit of recording decisions, failures, and benchmark results. Its weakest point was the same point that matters most for this new project: clean document intake. The old ingestion path treated PDF extraction as a mostly solved technical step, then spent most of its effort downstream on retrieval, answer behavior, claim extraction, graph pilots, and UI hardening.

Do not copy the old Phase 1 as-is. Its first executable path used pdfplumber.extract_text(), light line normalization, heuristic paragraph splitting, and later word-count chunking. That is not enough for a system whose value depends on complete and clean document organization.

What They Did Right

AreaUseful lessonEvidence in old project
Durable memory They kept project history, lessons, analysis reports, governance files, and run manifests. This made failures inspectable. PROJECT_HISTORY.md, platform/lessons.jsonl, docs/CURRENT_STATE.md, logs/run_manifest.jsonl
Bounded experiments Several phases had one-tuning-iteration caps and explicit gates. This prevented endless prompt bargaining. Phase 3a claims, Phase 4a entities, Phase 4b relations
Evaluation maturity They learned to distinguish dev benchmarks, trusted benchmarks, deterministic judge contracts, and noisy LLM judging. Phase 2.2, Phase 5 parity and metric work
Provenance pressure Claims and entities carried source references and grounding checks. This is worth keeping. src/extract_claims.py, src/extract_entities.py
Cost lessons They eventually learned to sample benchmark regions and match model tier to task shape instead of blindly running full-book expensive passes. Phase 3b Meditations reduced scope, Phase 4a model-tier refinement

What They Did Wrong Or Too Late

ProblemWhy it hurtNew-project response
PDF extraction was thin Raw page text and heuristic paragraphs became the substrate for chunking, retrieval, claims, and graph work. Build a multi-pass extraction audit before any retrieval work. Raw, cleaned, and canonical layers must be separate.
Document objects were not first-class enough Tables, figures, captions, footnotes, headers, page numbers, front matter, and commentary layers were treated mostly as noise or later retrieval hazards. Represent every detected content object explicitly with type, page span, coordinates when available, confidence, and review status.
One chunk object served too many jobs The project later learned that retrieval units and citation/provenance units may need different shapes. Keep canonical document objects separate from retrieval packages from the beginning.
Automation got ahead of perfection The system tried to become a broad reusable engine while still learning how to clean one book properly. Use one book as the proving ground. No automated multi-book path until Phase 1 quality gates pass.
Model selection was reactive Model choices were often discovered through runs instead of a written model policy with escalation rules. Create a model-use ledger before expensive model calls: deterministic first, cheap model for low-risk classification, stronger model for high-risk extraction review.
Folder layout mixed source, generated artifacts, tests, and experiments Book outputs and test fixtures accumulated across root-level artifact folders, making repo truth harder to scan. Separate engine code, documentation, book inputs, per-run artifacts, and experiments clearly.

Most Important Old Finding For Us

The old project eventually discovered that mechanically cleaner chunks were not automatically better retrieval chunks. That is true, but it starts even earlier: mechanically extracted text is not automatically clean document content. Phase 1 must preserve enough evidence to let us inspect and correct extraction before it becomes retrieval truth.

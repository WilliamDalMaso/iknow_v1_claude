Phase 1 Strategy

Date: 2026-05-27

Phase 1 Goal

Convert one PDF into a complete, clean, organized, auditable document package. The package must be good enough that later retrieval, reasoning, and knowledge graph work can trust its boundaries and provenance.

Non-Negotiable Gates

  1. Every PDF page is accounted for, including blank, image-only, front matter, back matter, and appendix pages.
  2. Raw extraction is preserved unchanged before cleanup.
  3. Clean text is derived from raw extraction through recorded operations, not silent rewriting.
  4. Headers, footers, page numbers, hyphenation fixes, OCR repairs, and dropped boilerplate are logged.
  5. Document objects are typed before retrieval: paragraph, heading, table, figure, caption, footnote, list, quote, code block, page artifact, unknown.
  6. Canonical reading order is explicit and inspectable.
  7. No content is deleted without a reviewable reason.
  8. One-book quality passes before multi-book automation begins.

Layer Model

  1. Source layer: original PDF plus file hash and metadata.
  2. Forensic layer: page dimensions, text availability, images, tables, fonts when available, OCR need, and extraction risk.
  3. Raw layer: raw text and layout output from extraction tools.
  4. Object layer: page-level content objects with type, order, provenance, confidence, and coordinates when available.
  5. Clean layer: normalized readable text by object, with reversible cleanup logs.
  6. Candidate reconstruction layer: provisional book structure, reading order, sections, page mappings, and content inventory.
  7. Canonical layer: promoted book structure and text after audit, comparison, and review.
  8. Audit layer: coverage reports, suspicious gaps, sampled visual checks, and human review notes.

Model Policy

Phase 1 should be deterministic-first. Models can assist classification, ambiguous repair, and review, but they should not be the first extractor of record. Expensive models require a written reason: high ambiguity, high value, or review of a specific failure. Cheap models can be used for low-risk labels only after deterministic evidence is gathered.

Candidate Lifecycle

Phase 1 v2 uses a candidate-first lifecycle: raw objects become classified candidate streams, candidate streams become a reconstruction map candidate, and only audited candidates can later become canonical outputs. This prevents clean-looking extraction from being treated as truth too early.

  1. raw_pages.jsonl and layout_objects.jsonl preserve source evidence.
  2. main_paragraph_candidates.jsonl holds body paragraph candidates.
  3. structure_candidates.jsonl holds headings, titles, section markers, and other meaningful non-body candidates.
  4. page_artifacts_candidates.jsonl holds likely page numbers, running headers, and non-content noise.
  5. unknown_objects.jsonl holds anything uncertain instead of deleting it.
  6. reconstruction_map_candidate.json links the candidates back to pages and source objects.

Paragraph Promotion Gate

After candidate streams are auditable, Phase 1 may promote paragraph candidates into canonical_paragraphs.jsonl. This first gate is intentionally narrow: it promotes paragraph candidates only, keeps structure unpromoted, and records every blocked object in promotion_blockers.jsonl. Canonical means evidence-bound and safe under current rules, not flawless.

  1. Promoted paragraphs must trace back to a candidate object, source object ids, source line ids, raw text, and clean text.
  2. Blocking warnings prevent promotion until reviewed or repaired.
  3. Non-paragraph streams are blocked from paragraph promotion instead of silently ignored.
  4. canonical_promotion_report.json records counts, warning policy, and validation status.

Canonical Paragraph Review

Promotion proves a paragraph passed the current gate; it does not prove the promoted layer is clean enough for retrieval or graph work. Phase 1 therefore reviews canonical_paragraphs.jsonl into canonical_paragraph_review_report.json before downstream intelligence can depend on it. This review flags risks but does not demote rows by itself.

  1. Look for likely merge problems, structure leakage, front-matter boundaries, unusual endings, and suspicious source spans.
  2. Record warning categories, severity, risk clusters, affected pages, risky samples, source candidate ids, source lines, and bounding boxes.
  3. Recommend the highest-impact risk to fix first before changing extraction logic.
  4. Use safe_for_downstream as a gate signal before retrieval, embeddings, reasoning, or graph work.

Page Furniture Detection

Phase 1 v3 adds candidate-only page furniture detection before any OCR or model review. The purpose is to prevent repeated running titles, page numbers, headers, and footers from being mistaken for book structure. These objects are not deleted; they move into page_artifacts_candidates.jsonl with evidence and remain inspectable in the audit.

  1. Normalize object text and strip attached page-number tokens.
  2. Look for repeated normalized furniture text across pages.
  3. Require margin position evidence near the top or bottom of the page.
  4. Require repeated nearby vertical position for strong running-header/footer candidates.
  5. Keep all exclusions candidate-only until audit promotion.

Review Gate

Detection alone is not trust. Page furniture candidates must pass a false-positive review before they can become canonical exclusions. The audit should make it easy to filter candidate objects, compare repeated patterns, and inspect risky cases such as front matter, appendix, chapter boundaries, small page ranges, and low repetition counts.

Review Overrides

Curated reviewer corrections live in reviews/<book_id>/review_overrides.jsonl. During extraction, Phase 1 replays those rows into data/runs/<book_id>/<run_id>/review_overrides_applied.jsonl. Overrides can move an object between candidate buckets while preserving the original detector assignment. They do not create canonical content and they do not change the detector itself. This makes review reusable without hiding the evidence trail.

Visual Page Witness

Phase 1 renders each source PDF page into page_images/ inside the generated run. These images are visual evidence only. They let a reviewer compare the original page, extracted objects, detector bucket, override bucket, and final candidate bucket without adding OCR, model review, retrieval, graph work, or canonical promotion.

Bounding-Box Overlay Witness

The audit draws existing extractor bounding boxes over each rendered page image. Object cards and overlay boxes share object ids, so a reviewer can click or hover between the visual location and the extracted object evidence. The overlay is evidence-only and does not change classification. Overlay controls may hide/show boxes by bucket, subtype, override state, or current filters to reduce visual noise on dense pages.

First Book Strategy

We will choose one book and build Phase 1 around it until it passes cleanliness gates. This is intentionally slower than broad automation. The first success condition is not throughput. It is trust in the extracted document.

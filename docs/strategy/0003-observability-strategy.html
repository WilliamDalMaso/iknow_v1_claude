Observability Strategy

Date: 2026-05-27

Observability should make the extraction process inspectable while it is being built. It should show evidence, run state, and quality risks, not just activity.

Current Capabilities

  1. Local black-and-white dashboard at http://127.0.0.1:8765.
  2. Append-only runtime events in ignored JSONL.
  3. Phase 1 run summaries discovered from data/runs/*/phase1_v0/.
  4. Links to generated Phase 1 audit pages and source manifests through localhost.

What Observability Should Track Next

  1. Page coverage: total pages, text pages, image-only pages, blank/unreadable pages.
  2. Object coverage: headings, paragraphs, page artifacts, unknown objects, tables, figures, footnotes.
  3. Extraction risk: CID noise, missing text, OCR-needed pages, suspiciously short pages.
  4. Cleanup risk: operations applied, objects changed, objects needing review.
  5. Gate status: whether Phase 1 acceptance checks pass or fail.

Boundary

Runtime observability logs are not durable project history. When an observation matters, promote it into docs/history/, docs/lessons/, or docs/strategy/.
